# Database Schema Design - Kpop News Digest


---

## Overview

This schema defines a **minimal and efficient database structure** for the Kpop News Digest project.  
It supports articles aggregated from various APIs (e.g., Naver, Daum, RSS) and associates them with multiple artists (e.g., BTS, BLACKPINK).

---

## Prisma Schema

```prisma
model Article {
  id            String   @id @default(cuid())
  originalUrl   String   @unique          // Original article URL (as received)
  finalUrl      String?                   // Redirected final URL (if available)
  title         String?                   // AI-generated rewritten title
  titleRaw      String                    // Original or literal English title
  publishedAt   DateTime
  api           String                    // Source API name (e.g., "naver", "daum", "rss-yonhap")
  source        String                    // Actual news outlet name (e.g., "Yonhap News")
  summary       String?                   // AI-generated summary
  createdAt     DateTime  @default(now()) // Timestamp when the article was added
  updatedAt     DateTime  @updatedAt

  // Many-to-many relationship with artists
  artists       ArticleArtist[]

  @@index([publishedAt])                  // Optimize latest-news queries
}

model Artist {
  id        String   @id @default(cuid())
  name      String   @unique              // Display name (e.g., "BLACKPINK")
  slug      String   @unique              // URL slug (e.g., "blackpink")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  articles  ArticleArtist[]               // Related articles (virtual relation, no duplication in DB)
}

model ArticleArtist {
  id         String   @id @default(cuid())
  article    Article  @relation(fields: [articleId], references: [id])
  articleId  String
  artist     Artist   @relation(fields: [artistId], references: [id])
  artistId   String
}
```

---

## Notes for Agent

### 1️⃣ Relationship Summary
- **Article ↔ Artist** = Many-to-Many via `ArticleArtist`
- A single article can be linked to multiple artists.
- Each artist can have multiple related articles.

### 2️⃣ Index Explanation
- `@@index([publishedAt])` → Speeds up sorting by newest articles.

### 3️⃣ API and Source
- `api` identifies the **fetching method or provider** (e.g., `"naver"`, `"daum"`, `"rss-yonhap"`).
- `source` is the **actual media outlet name** that published the article (e.g., `"The Korea Times"`).

### 4️⃣ Optional Fields
- `finalUrl` is stored when redirects occur (useful for deduplication or link checking).
- `summary` and `title` are generated by LLM-based summarization and rewriting modules.

### 5️⃣ Artist Page Logic
- `/` → Show all news (`Article` table ordered by `publishedAt DESC`)
- `/bts` → Query articles where `artists.some.artist.slug == "bts"`
- `/blackpink` → Same pattern for other artists

### 6️⃣ Minimal Query Examples

```ts
// Get latest 50 news articles
const allNews = await prisma.article.findMany({
  orderBy: { publishedAt: "desc" },
  take: 50,
});

// Get latest 50 BTS-related news
const btsNews = await prisma.article.findMany({
  where: { artists: { some: { artist: { slug: "bts" } } } },
  orderBy: { publishedAt: "desc" },
  take: 50,
});
```

## Summary

| Table | Purpose | Key Fields |
|--------|----------|-------------|
| **Article** | Stores all fetched articles | `originalUrl`, `api`, `source`, `publishedAt`, `createdAt` |
| **Artist** | Registered K-pop artists | `name`, `slug` |
| **ArticleArtist** | Connects articles ↔ artists | `articleId`, `artistId` |

---

**This schema is intentionally minimal**: no extra status enums, alias tables, or unique constraints beyond essentials.
